# 카드 명세서 분석 프로그램: 최종 개발 계획서

## 1. 프로젝트 개요

본 문서는 사용자의 모든 요구사항을 통합하고, **거래별 메모 및 태그 기능**을 추가하며, 프로그램의 최적 배포 형태(로컬 vs. 웹)에 대한 분석과 추천을 담은 **최종 개발 계획서**입니다. 이 계획서는 개인 맞춤형 재정 관리 도우미 개발을 위한 포괄적인 청사진을 제공합니다.

## 2. 최종 기능 명세

- **사용자 주도적 분류**: 사용자가 직접 소비 카테고리를 생성, 수정, 삭제하며, 가맹점별로 카테고리를 지정합니다.
- **영속적 데이터 관리**: 사용자가 설정한 카테고리와 가맹점 분류 정보는 로컬 데이터베이스(SQLite)에 영구 저장되어, 프로그램 재시작 또는 명세서 추가 시에도 그대로 유지됩니다.
- **거래별 상세 관리 (신규 기능)**: 모든 거래 내역에 개별적으로 **메모**를 작성하고, 자유롭게 **태그**(#업무, #휴가 등)를 추가하여 지출의 구체적인 맥락을 기록하고 검색할 수 있습니다.
- **다차원 분석 리포트**: 월별/연도별 리포트에서 카테고리별 분석은 물론, 태그별 지출 내역을 필터링하고 집계하는 기능을 제공합니다.
- **직관적 UI/UX**: 별도의 카테고리 관리 페이지와 거래 내역에서 직접 메모/태그를 편집할 수 있는 편리한 인터페이스를 제공합니다.

## 3. 로컬 프로그램 vs. 웹 애플리케이션: 비교 및 추천

프로그램의 배포 형태에 대해 장단점을 비교 분석한 결과, 본 프로젝트의 성격에 가장 적합한 방식을 다음과 같이 추천합니다.

| 구분 | 로컬 프로그램 (권장) | 웹 애플리케이션 |
| :--- | :--- | :--- |
| **보안/개인정보** | **최상**. 금융 정보가 사용자 컴퓨터를 절대 벗어나지 않아 가장 안전합니다. | **우려**. 서버에 데이터를 전송해야 하므로 해킹 및 유출의 위험이 존재하며, 사용자의 신뢰가 필수적입니다. |
| **접근성** | 제한적. 프로그램이 설치된 컴퓨터에서만 사용 가능합니다. | **우수**. 인터넷만 되면 어디서든 스마트폰, 태블릿 등으로 접근 가능합니다. |
| **유지보수/업데이트** | 수동. 사용자가 직접 새 버전을 다운로드하여 설치해야 합니다. | **자동**. 개발자가 서버만 업데이트하면 모든 사용자가 즉시 새 기능을 사용합니다. |
| **설치/설정** | 필요. 최초 1회 설치 과정이 필요합니다. | **불필요**. 웹사이트 주소로 접속만 하면 바로 사용할 수 있습니다. |
| **비용** | **사용자 무료**. 서버 운영 비용이 없어, 한 번 개발하면 추가 비용이 발생하지 않습니다. | **지속 발생**. 서버, 도메인 등 유지비가 필요하며, 이로 인해 유료 구독 모델이 될 수 있습니다. |

### **최종 추천: 로컬 프로그램**

**결론적으로 '로컬 프로그램' 방식을 강력하게 추천합니다.** 사용자의 민감한 금융 정보를 다루는 만큼, 데이터가 외부로 전송될 일 없는 로컬 방식이 보안 측면에서 압도적으로 우수합니다. 여러 기기에서의 접근성이라는 장점보다는, **개인정보보호**라는 핵심 가치를 지키는 것이 이 프로그램의 정체성에 더 부합합니다. 사용자가 자신의 PC에서 안심하고 가계부를 관리하는 경험을 제공하는 것을 최우선 목표로 삼아야 합니다.

## 4. 최종 아키텍처 및 개발 계획

거래별 메모/태그 기능 추가를 위해 데이터베이스 스키마를 확장하고, 전체 개발 계획을 다음과 같이 최종 확정합니다.

### 데이터베이스 스키마 확장

- **`transactions` 테이블**: `id`, `date`, `merchant`, `amount`, `category_id` 등 기본 거래 정보
- **`categories` 테이블**: `id`, `name` 등 사용자 정의 카테고리 정보
- **`memos` 테이블 (신규)**: `id`, `transaction_id`, `content` 등 거래별 메모 정보
- **`tags` 테이블 (신규)**: `id`, `name` 등 생성된 모든 태그 정보
- **`transaction_tags` 테이블 (신규)**: `transaction_id`, `tag_id`를 연결하는 중간 테이블

### 최종 개발 단계 및 일정

| 단계 | 주요 과업 | 예상 기간 |
| :--- | :--- | :--- |
| **1단계: 환경 및 DB 설계** | - 개발 환경 구축, 최종 DB 스키마(메모/태그 포함) 설계<br>- 데이터베이스 연결 및 테이블 생성 | 3일 |
| **2단계: 핵심 데이터 처리** | - 명세서 파싱 및 DB 저장<br>- DB의 분류 규칙에 따른 카테고리 자동 적용 로직 구현 | 3일 |
| **3단계: 카테고리 및 태그 관리** | - 카테고리 관리(CRUD) API 및 UI 개발<br>- 거래 내역에 메모/태그를 추가/수정하는 API 및 UI 개발<br>- 태그 자동완성 기능 구현 | 6일 |
| **4단계: 분석 리포트 고도화** | - 월별/연도별 리포트 UI 구현<br>- 태그 기반 데이터 필터링 및 검색 기능 추가<br>- Plotly.js를 이용한 동적 차트 렌더링 | 4일 |
| **5단계: 패키징 및 배포** | - PyInstaller 등을 사용하여 단일 실행 파일(.exe, .app)로 패키징<br>- 설치 및 사용법을 담은 최종 사용자 매뉴얼 작성 | 3일 |

이 최종 계획서는 보안, 사용 편의성, 확장성을 모두 고려한 최적의 안입니다. 이 계획에 따라 개발을 진행하면, 사용자가 안심하고 사용할 수 있는 강력하고 실용적인 개인 재무 관리 프로그램을 만들 수 있을 것입니다.
